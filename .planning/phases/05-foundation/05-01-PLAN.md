  <action>
    Create `openhands/analytics/analytics_context.py` with:

    1. A `@dataclass` called `AnalyticsContext` with fields:
       - `user_id: str` -- the raw user ID string
       - `consented: bool` -- whether user has opted in to analytics (False = safe default)
       - `org_id: str | None` -- string org_id from user.current_org_id (None if missing)
       - `user: Any | None` -- the full User ORM object (None if lookup failed). Use `Any` to avoid importing enterprise types.

    2. An async factory function `resolve_context(user_id: str) -> AnalyticsContext` that:
       - Imports `UserStore` dynamically inside the function body (same pattern used in auth.py line 444: `from storage.user_store import UserStore`)
       - Calls `await UserStore.get_user_by_id(user_id)`
       - If user is None, returns `AnalyticsContext(user_id=user_id, consented=False, org_id=None, user=None)`
       - If user exists, extracts: `consented = user.user_consents_to_analytics is True` (None = undecided = not consented, same pattern as auth.py line 263)
       - Extracts: `org_id = str(user.current_org_id) if user.current_org_id else None`
       - Returns `AnalyticsContext(user_id=user_id, consented=consented, org_id=org_id, user=user)`
       - The entire body is wrapped in try/except Exception that logs via `openhands.core.logger.openhands_logger` and returns the safe default

    3. Update `openhands/analytics/__init__.py`:
       - Add imports: `from openhands.analytics.analytics_context import AnalyticsContext, resolve_context`
       - Add to `__all__`: `'AnalyticsContext'`, `'resolve_context'`

    Write tests FIRST in `tests/unit/test_analytics_context.py`. Mock `UserStore.get_user_by_id` using `unittest.mock.patch` targeting the dynamic import path. Follow the existing test style in test_analytics_service.py (class-based grouping, descriptive docstrings).

    IMPORTANT: analytics_context.py must NOT import from enterprise/ at module level. The `from storage.user_store import UserStore` import must be inside `resolve_context()` function body, matching the established pattern used throughout the codebase (e.g., auth.py line 444, conversation_callback_utils.py line 80).

    - Test 3: identify_user in SaaS mode with consent calls set_person_properties with email, org_id, org_name, idp, last_login_at
    - Test 4: identify_user in SaaS mode with consent and org_members calls group_identify for each org

  <action>
    Add an `identify_user` method to `AnalyticsService` in `openhands/analytics/analytics_service.py`:

    ```python
    def identify_user(
        self,
        distinct_id: str,
        consented: bool = True,
        email: str | None = None,
        org_id: str | None = None,
        org_name: str | None = None,
        idp: str | None = None,
        orgs: list[dict[str, Any]] | None = None,
    ) -> None:
    ```

    This method consolidates the duplicated pattern from auth.py (lines 452-499) and oauth_device.py (lines 334-355):

    1. Consent gate: return immediately if `not consented`
    2. SaaS gate: return immediately if `self._app_mode != AppMode.SAAS` (person profiles are SaaS-only)
    3. Call `self.set_person_properties()` with person properties dict: email, org_id, org_name, plan_tier (None -- deferred), idp, last_login_at (datetime.now(timezone.utc).isoformat())
    4. If `orgs` is provided, iterate and call `self.group_identify()` for each org. Each org dict has keys: `id` (str), `name` (str | None), `member_count` (int | None). Group properties: org_name, plan_tier (None), created_at (None), member_count.
    5. Wrap everything in try/except Exception, log via logger.exception('AnalyticsService.identify_user failed')

    The `orgs` parameter uses plain dicts (not ORM objects) so the service stays decoupled from enterprise/. The caller (auth.py, oauth_device.py) is responsible for building these dicts from ORM data -- but that migration happens in Phase 6, not here.

    Write tests FIRST in `tests/unit/test_analytics_service.py` as a new `TestIdentifyUser` class. Use the existing `saas_service` and `oss_service` fixtures. Verify that the existing 30 tests are not broken.
